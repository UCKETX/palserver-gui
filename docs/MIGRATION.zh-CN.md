# 存档迁移教程

[繁體中文](MIGRATION.md) | **简体中文**

把既有的 Palworld 世界搬进 palserver GUI 管理的实例。三种来源、同一套原则。

> **开始之前**:所有迁移都要在**服务器停止**的状态下进行,而且来源端的游戏/服务器也要完全关闭 —— Palworld 运行时会持续写入存档,拷贝到一半的文件是坏的。

---

## 存档结构

```
<服务器目录>/Pal/Saved/
├─ SaveGames/0/<世界 GUID>/          ← 一个世界一个文件夹
│  ├─ Level.sav                      世界本体(建筑、帕鲁、地形改变)
│  ├─ LevelMeta.sav
│  ├─ WorldOption.sav                创建世界时的设置快照
│  └─ Players/<PlayerUid>.sav        每位玩家的角色
└─ Config/WindowsServer/
   ├─ PalWorldSettings.ini           世界设置(由 GUI 产生,别手动改)
   └─ GameUserSettings.ini           ★ DedicatedServerName 决定加载哪个世界
```

**最重要的一件事**:服务器不会自动加载「唯一那个」世界文件夹,而是读
`GameUserSettings.ini` 里的 `DedicatedServerName=` 去找对应 GUID 的文件夹。
迁移失败九成是这里没对上 —— 服务器找不到,就默默开一个全新世界。

在 GUI 里,**存档备份**标签页会把每个世界的 GUID 和「激活中」状态直接列出来,
按「设为激活世界」就会帮你改好这一行,不用手动编辑。

---

## 情境 A:从别台专用服务器搬过来

最单纯,格式完全相同。

1. **来源端**:停止旧服务器。到 `Pal/Saved/SaveGames/0/`,找到你的世界文件夹
   (一串英数字的 GUID)。整个文件夹压缩成 zip。
2. **目标端**:在 GUI 创建实例后,**先启动一次再停止** —— Palworld 只有在第一次
   启动时才会生成 `SaveGames/0/` 和配置文件。
3. GUI → **模组**标签页 → 「浏览全部」→ 进到 `Pal/Saved/SaveGames/0/`,
   用「上传文件」把 zip 里的内容放进去(或用「新文件夹」建好 GUID 文件夹后
   上传里面的 .sav 档)。
4. GUI → **存档备份**标签页 → 找到刚上传的世界 → 按「**设为激活世界**」。
5. 启动服务器。玩家用原本的角色进来即可。

> **不要直接拷贝旧的 `GameUserSettings.ini` 过来**。它夹带了旧主机的
> `PublicIP`、`PublicPort`、RCON 设置,会让新服务器列表显示错误或远程管理失效。
> 只需要改 `DedicatedServerName` 这一行,而这件事 GUI 帮你做。

---

## 情境 B:从 v1 palserver-GUI 搬过来

v1 把每个服务器放在自己的文件夹里,底下的 `Pal/Saved/` 结构和情境 A 完全一样。

1. 在 v1 接口按「打开服务器文件夹」,进到 `Pal/Saved/SaveGames/0/`。
2. 之后照**情境 A 的第 1 步起**操作即可。

如果 v1 的服务器就在同一台机器上,更省事的做法是**直接收编**:在 GUI 创建实例时,
「既有服务器路径」填 v1 那个服务器目录(含 `PalServer.exe` 的那一层)。
不用搬任何文件,世界、模组、设置全部原地接管。

---

## 情境 C:从本机多人游戏(四人邀请码)搬过来

这是唯一会踩到**角色重置 bug** 的情境:本机存档的 PlayerUid 和专用服务器的
编码方式不同,直接搬过去,玩家进来会被要求创建新角色(帕鲁和建筑还在,
角色本身不见)。

1. 找到本机存档:
   `%LOCALAPPDATA%\Pal\Saved\SaveGames\<你的 SteamID>\<世界 GUID>\`
2. 照情境 A 上传到服务器的 `SaveGames/0/` 下。
3. **修正玩家存档**:用社群工具
   [palworld-host-save-fix](https://github.com/xNul/palworld-host-save-fix)
   把主机玩家的 `.sav` 转成专用服务器格式。你需要准备:
   - 旧的 host 存档文件名(通常是 `00000000000000000000000000000001.sav`)
   - 每位玩家新的 PlayerUid(让他们先各自进服务器一次创建空角色,
     GUI 的**存档备份**标签页会列出新产生的玩家存档文件名)
4. 回到 GUI,「设为激活世界」→ 启动。

> 没有工具能百分之百保证主机角色转移成功。**动手前先在 GUI 按「立即备份」**,
> 出事可以一键还原。

## 迁移后的检查清单

- [ ] **存档备份**标签页里,你的世界标示为「激活中」
- [ ] 世界大小合理(空世界通常不到 1 MB;玩过的世界几十 MB 起跳)
- [ ] 玩家存档数量符合预期
- [ ] 启动后日志没有 `Save data is corrupted`
- [ ] 进游戏确认建筑、帕鲁、等级都在

---

## 常见问题

**进去变成全新世界,旧的东西都不见了**
`DedicatedServerName` 没指到你的 GUID。到**存档备份**标签页按「设为激活世界」。

**世界在,但要我重建角色**
情境 C 的 PlayerUid 问题,见上方第 3 步。若是从专用服务器搬到专用服务器,
检查 `Players/` 文件夹是不是漏掉没拷贝。

**启动就 `Save data is corrupted` 然后崩溃**
存档在拷贝时是开着的(来源端没关干净),或传输过程截断。用备份还原,
重新从来源端拷贝一次。

**跨平台(Windows 服务器 ↔ Linux 服务器)可以吗?**
可以,`.sav` 格式相同。注意配置文件在 `Config/WindowsServer/` 与
`Config/LinuxServer/` 两个不同文件夹,GUI 会依 agent 所在平台自动处理。

---

## 顺带一提:自动备份

迁移完成后,建议到**存档备份**标签页把**自动备份**打开。agent 会依你设置的间隔,
在服务器运作时定期打包激活中的世界,超过保留份数的旧备份自动清除;
勾了「没有玩家在在线时跳过」就不会堆积一堆一模一样的文件。

备份前 agent 会先请服务器把世界写入磁盘(通过 REST API),所以拿到的不是
写到一半的快照 —— 前提是 `RESTAPIEnabled` 有开、`AdminPassword` 有设。
